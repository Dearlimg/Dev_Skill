# Skill 创建与优化规范

## 核心定位

创建高质量 skill，并通过测试评估持续迭代优化。适用于从零创建、编辑改进、效果评估、触发率优化等场景。

---

## 七步迭代循环

```
1.确定目标 → 2.编写草稿 → 3.创建测试用例 → 4.运行对比测试 → 5.评估结果 → 6.迭代改进 → 7.扩大规模
                                                                        ↑________________________|
                                                                          (重复 2-6 直到达标)
```

1. **确定技能目标**：明确 skill 做什么、何时触发、如何做
2. **编写草稿**：撰写 SKILL.md（见下文规范）
3. **创建测试用例**：2-3 个真实用户会说的提示词
4. **运行对比测试**：同时跑「有 skill」和「无 skill / 旧 skill」基线对照
5. **评估结果**：定性（人工查看）+ 定量（断言评分）
6. **迭代改进**：根据反馈重写，重复测试
7. **扩大规模**：扩充测试集，再次验证泛化能力

## Skill 文件结构规范

```
skill-name/
├── SKILL.md          # 必需：前置元数据 + 指令正文
├── scripts/          # 可选：重复性任务的执行脚本
├── references/       # 可选：按需加载的参考文档
└── assets/           # 可选：输出用的模板/图标/字体
```

### 三级渐进式加载

| 级别 | 内容 | 加载时机 | 理想长度 |
|------|------|---------|---------|
| 1 | 元数据（name + description） | 始终在上下文 | ~100 词 |
| 2 | SKILL.md 正文 | skill 触发时加载 | < 500 行 |
| 3 | 捆绑资源 | 按需加载 | 无限制 |

> 主文件保持精简，大段内容放 `references/`，避免一次性撑爆上下文。

## description 编写要点

description 是触发的关键，必须包含**做什么 + 何时使用**。

- 写得**略带推动性**：模型倾向「欠触发」，description 要主动覆盖相关场景
- 所有「何时使用」信息放 description，而非正文
- 描述具体触发词和场景，而非泛泛而谈

**对比示例：**
- ❌ 普通：`"代码审查规范"`
- ✅ 推动：`"代码审查规范：以资深架构师视角逐层审查架构、逻辑、安全、性能等维度，分级反馈问题并给出具体修改建议。当用户提交代码、请求 review、检查代码质量时触发"`

---

## 写作风格

- **祈使句**编写指令（「检查边界条件」而非「你应该检查...」）
- **解释「为什么」**而非堆砌 MUST/NEVER，让模型理解原理
- 相信模型的理解力，用理论思维而非死板规则
- 写完草稿后用「新鲜眼光」审视，移除不产生价值的内容
- 避免过拟合到具体示例，追求通用性

## 测试评估

### 测试用例结构

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "用户真实会说的提示词",
      "expected_output": "预期结果描述"
    }
  ]
}
```

### 评估要求

- 用真实用户口吻写 prompt，不要抽象表述
- 同时跑「有 skill」和「无 skill」对照，才能看出增量价值
- 定性看输出质量，定量用断言评分

### description 触发率优化

生成 20 个评估查询：
- **should-trigger**（8-10 个）：覆盖不同表达方式，含正式/随意/边缘场景
- **should-not-trigger**（8-10 个）：近义词但实际不需要该 skill 的「近邻陷阱」

> 查询要具体真实，不要 `"格式化数据"` 这种抽象句，要 `"老板发了个 xlsx 让我加一列利润率百分比..."` 这种真实口吻。

## 八荣八耻（Skill 编写准则）

| 以…为耻 | 以…为荣 |
|---------|---------|
| description 太抽象 | 具体推动性描述 |
| 正文堆砌 MUST/NEVER | 解释为什么 |
| 一次性塞满上下文 | 渐进式分级加载 |
| 过拟合具体示例 | 追求通用泛化 |
| 不测就交付 | 对比测试验证增量 |
| 模糊测试用例 | 真实用户口吻 |
| 保留冗余内容 | 移除无价值段落 |
| 一次定稿不改 | 迭代循环改进 |
