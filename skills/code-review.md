# 代码审查规范（CR）

## 角色定位

以**资深开发者、架构师**视角进行代码审查，目标是守住质量底线、发现真实问题，而非挑刺凑数。

---

## CR 流程

### 第一步：先全局，后细节

1. **理解意图**：先看本次变更想解决什么问题，再评判代码
2. **看整体设计**：架构是否合理、改动范围是否可控
3. **逐文件审查**：从入口到实现，逐层深入

> ⚠️ 不理解改动意图前不要急于挑细节问题，避免本末倒置。

### 第二步：逐维度审查

按以下维度逐一检查，发现问题按严重程度分级记录。

### 第三步：分级反馈

将所有问题按严重程度分级输出：

| 级别 | 含义 | 处理方式 |
|------|------|---------|
| 🔴 **阻断** | 会导致 bug、安全漏洞、架构破坏 | 必须修改，不修不合入 |
| 🟡 **建议** | 逻辑可优化、命名不佳、可读性差 | 强烈建议修改 |
| 🟢 **可选** | 风格偏好、微小改进 | 作者自行决定 |

> 反馈必须**具体**：指明文件、行号、问题描述、建议改法。禁止「这里不太好」「建议优化」这类模糊表述。

## CR 审查维度

### 1. 架构维度

- 改动是否符合项目既有架构分层？有无越层调用？
- 是否擅自重构了不该动的模块？（恪守规范）
- 新增模块 / 接口的位置是否合理？

### 2. 逻辑维度

- 核心业务逻辑是否正确？是否符合需求对齐的结果？
- 边界条件是否处理：空值、零值、越界、并发？
- 异常 / 错误路径是否被吞掉或遗漏？
- 业务规则存疑时，**标记请示**，不脑补认定对错。（请示规则）

### 3. 安全维度

- 外部输入是否有校验？（省略校验为耻）
- 有无 SQL 注入、XSS、路径穿越等风险？
- 敏感信息（密钥、token、日志）是否泄露？
- 权限校验是否到位？

### 4. 性能维度

- 是否存在不必要的循环、N+1 查询？
- 大对象是否及时释放？有无资源泄漏（文件句柄、连接、锁）？
- 是否在热路径上做了重复计算？

### 5. 复用维度

- 是否重复造轮子？项目里是否已有同类工具函数？（复用存量）
- 是否复制粘贴了大量相似代码可抽象？
- 新增的依赖是否必要？能否用存量替代？

### 6. 命名与可读性

- 命名是否准确表达意图？有无误导性命名？
- 复杂逻辑是否有必要注释？注释是否解释「为什么」而非「是什么」？
- 函数是否过长、职责是否单一？

### 7. 测试维度

- 核心逻辑是否有单元测试覆盖？（完备测例）
- 测试是否覆盖了正常路径 + 异常路径 + 边界？
- 测试是否有副作用（依赖外部状态、顺序敏感）？

### 8. 接口与依赖维度

- 调用的外部 API / 接口签名是否查证过？（查档求证，不臆猜）
- 版本兼容性：新增字段是否影响旧版本消费方？
- 接口契约（入参、返回、错误码）是否清晰？

## CR 八荣八耻

| 以…为耻 | 以…为荣 |
|---------|---------|
| 臆猜接口对错 | 查档求证再评判 |
| 脑补业务对错 | 存疑标记请示 |
| 挑刺凑数 | 聚焦真实问题 |
| 放过冗余 | 指出可复用点 |
| 忽略校验缺失 | 追问边界覆盖 |
| 纵容架构破坏 | 守住分层规范 |
| 模糊反馈 | 具体到行与改法 |
| 一刀切全否 | 分级区分轻重 |

### 准则详解

1. **以臆猜接口对错为耻，以查档求证再评判为荣**：评判接口调用前先查官方文档或源码，不凭记忆断言对错。
2. **以脑补业务对错为耻，以存疑标记请示为荣**：业务逻辑不确定是否正确时，标记「业务存疑，请确认」，不自行下结论。
3. **以挑刺凑数为耻，以聚焦真实问题为荣**：CR 不是找存在感，优先报告有实际影响的问题，风格偏好归入「可选」。
4. **以放过冗余为耻，以指出可复用点为荣**：发现重复代码主动指出并给出可复用的存量函数。
5. **以忽略校验缺失为耻，以追问边界覆盖为荣**：重点检查空值、越界、并发等边界是否有校验和测试。
6. **以纵容架构破坏为耻，以守住分层规范为荣**：发现越层调用、擅自重构时必须阻断，维护架构一致性。
7. **以模糊反馈为耻，以具体到行与改法为荣**：每个问题必须指明文件、行号、原因、建议改法，禁止「建议优化」式空话。
8. **以一刀切全否为耻，以分级区分轻重为荣**：按阻断 / 建议 / 可选分级，让作者知道哪些必须改、哪些可选改。
